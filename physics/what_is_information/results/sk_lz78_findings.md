## 1. What was measured

Two computable lower-bound proxies for Kolmogorov complexity K were run against 16 string archetypes:

- **gzip_ratio** — compressed_size / original_size using LZ77 sliding-window (compresslevel=9). Exploits local back-references within a ~32KB window.
- **lz78_norm** — phrase_count / (n / log2(n)) from the LZ78 factorization. Builds a global dictionary incrementally; no window limit, but each phrase is added to the dictionary exactly once.

Both are lower bounds: if either compresses the string, K is provably bounded above by the compressed size. If neither compresses it, K may still be small (globally-algorithmic strings).

---

## 2. Critical normalization finding: LZ78 is alphabet-dependent

The normalization `lz78_norm = phrase_count / (n / log2(n))` assumes a binary alphabet. For a random string over a |Sigma|-symbol alphabet, the expected phrase count is `n / log_{|Sigma|}(n) = n * log2(|Sigma|) / log2(n)`. For bytes (|Sigma|=256, log2(256)=8), the expected random value is `8n/log2(n)`, not `n/log2(n)`.

Measured confirmation:

| String | Alphabet | lz78_norm |
|---|---|---|
| random_bytes (256 symbols) | 256 | 6.70 |
| base64_random (64 symbols) | 64 | 5.63 |
| dna_random (4 symbols) | 4 | 2.55 |

The ratio 6.70 : 5.63 : 2.55 is close to log2(256) : log2(64) : log2(4) = 8 : 6 : 2. The systematic scaling confirms the formula. **LZ78 phrase counts must be normalized by `n * log2(|Sigma|) / log2(n)` to be alphabet-invariant.** This matters for R1: any physical K bound derived from LZ78 is alphabet-relative, not absolute.

## 3. Agreement zone: where both proxies reliably signal K

Only three archetypes land in AGREE-HIGH (both gzip and LZ78 see incompressible structure):

| Label | gzip | lz78_norm | True K |
|---|---|---|---|
| random_bytes | 1.0023 | 6.70 | K=MAX |
| base64_random | 0.7592 | 5.63 | K=HIGH |
| lz77_adversarial (LCG) | 1.0023 | 6.70 | K=medium (short program!) |

The first two are genuinely high-K (no program computes them). The third is the key failure: the LCG string has a short generating program (~20 bytes) but both proxies classify it as maximally complex. This is where both lower bounds are simultaneously loose.

**Conclusion for R1**: In the agree-high zone, the lower bound `K ≥ n * (1 - epsilon)` is tight. Physical systems realizing these strings in n bits genuinely require approximately n bits of description. The holographic bound applies cleanly here.

## 4. Divergence zone: where gzip and LZ78 disagree

### 4a. Structured / periodic strings (LZ78 > gzip)

| Label | gzip | lz78_norm | g/l ratio |
|---|---|---|---|
| constant_zeros | 0.0045 | 0.19 | 0.024 |
| period_2 | 0.0046 | 0.26 | 0.017 |
| period_16 | 0.0070 | 0.74 | 0.009 |
| fibonacci_word | 0.0109 | 0.65 | 0.017 |
| thue_morse | 0.0132 | 0.88 | 0.015 |

Gzip exploits period and repetition via back-references far more efficiently than LZ78. A period-2 string ABABAB... has constant gzip output size (the back-reference mechanism re-uses the same 2-byte pattern indefinitely within the window), but LZ78 must add a new phrase every time it sees a symbol-sequence extension not yet in the dictionary. For period_16, LZ78 phrase count = 558, producing lz78_norm = 0.74 — classified as high-K by LZ78 even though the string is trivially generated by a 16-byte cycle.

**Takeaway**: gzip is the better K-proxy for periodic and locally-repetitive strings. LZ78 overestimates K here. The g/l ratio (0.009–0.024) is the signature of strong local periodicity that LZ78 cannot exploit.

### 4b. Natural language and source code (LZ78 >> gzip)

| Label | gzip | lz78_norm | g/l ratio |
|---|---|---|---|
| english_prose | 0.0435 | 2.80 | 0.016 |
| source_code_python | 0.0406 | 2.54 | 0.016 |

These are the most extreme divergences. Gzip compresses natural language and code to ~4% of original size. LZ78 barely touches them (lz78_norm = 2.5–2.8, meaning the LZ78 phrase count is 2.5–2.8 times what you'd expect for random binary). The reason: natural language is full of repeated phrases ("the", "of", "import", "def"), which gzip's back-reference window exploits every occurrence of, but LZ78 can only add each phrase to its dictionary once. After the first 1000 tokens, new occurrences of known words do get matched, but only the exact previously-seen extension. LZ78's dictionary grows monotonically and never re-uses entries in the way gzip's window re-scans do.

**Takeaway**: For human-generated structured content, gzip is a much tighter lower bound on K than LZ78. The true K for the prose passage (repeated Shannon 1948 abstract) is O(len(passage)) ≈ 400 bytes, and gzip correctly gets close to this. LZ78 gives a loose bound 64× larger.

### 4c. Math constants (both fail, but in a quantifiable way)

| Label | gzip | lz78_norm | g/l ratio | True K |
|---|---|---|---|---|
| pi_digits | 0.5085 | 3.65 | 0.1392 | O(1) |
| e_digits | 0.5084 | 3.65 | 0.1391 | O(1) |
| sqrt2_digits | 0.5082 | 3.65 | 0.1393 | O(1) |
| random_bytes (ref) | 1.0023 | 6.70 | 0.1496 | K=MAX |

The three math constants produce nearly identical (gzip, lz78) pairs: gzip ≈ 0.508, lz78_norm ≈ 3.65. The g/l ratio ≈ 0.139 vs 0.150 for true-random. **The relative difference is only 7%.**

This is the central result of this scan: both proxies fail identically on globally-algorithmic strings. The g/l ratio is a fingerprint of "random-looking digit expansion" — whether the true K is O(1) (math constants) or O(n) (truly random), both proxies produce the same g/l ratio to within 7%. Neither proxy can distinguish π from noise.

The 7% gap is not noise: every math constant gives g/l = 0.139 ± 0.0001, while random bytes give 0.150 ± 0.001. The difference comes from the 10-symbol digit alphabet: digit strings are over {0..9} (|Sigma|=10), so their LZ78 phrase counts are slightly lower than byte-random, and gzip also gets marginally more compression. The alphabet effect explains the systematic offset.

### 4d. RLE runs (mixed structure)

| Label | gzip | lz78_norm | g/l ratio |
|---|---|---|---|
| rle_random_runs | 0.1565 | 2.95 | 0.053 |

Gzip exploits the local runs (repetition within a run maps to back-references), compressing to 15.6%. LZ78 sees the global picture of distinct symbol sequences, producing lz78_norm = 2.95. The g/l = 0.053 is intermediate between the structured zone (0.009–0.024) and the random zone (0.13–0.15), correctly placing RLE in the "locally structured, globally random" middle ground.

## 5. Summary: the g/l ratio as a K-diagnostic

The ratio `g/l = gzip_ratio / lz78_norm` partitions the archetypes into three bands:

| g/l range | Regime | Examples |
|---|---|---|
| 0.009–0.024 | Locally periodic / highly structured | constant, period_2, period_16, fibword |
| 0.015–0.016 | Natural structured content | english prose, source code |
| 0.053 | Mixed local structure | RLE runs |
| 0.126–0.150 | Random-like (high K or globally-algorithmic) | random, base64, DNA, π, e, √2, LCG |

The math constants land in the random-like band (0.139) rather than the structured band, confirming that both proxies are blind to global algorithmic structure. The LCG adversarial string also lands at 0.150 — indistinguishable from true random.

## 6. Implications for R1 (bounding K in physical regions)

**R1 asks**: what physical quantities bound K-information in a region?

The numerical results sharpen the question:

**Finding 1 — the lower bound gap**: Both gzip and LZ78 give K-lower-bounds that are provably too loose for globally-algorithmic strings. The gap is not small: the LCG string has a ~20-byte generating program but both proxies assign K ≈ 10,000 bits. The lower bound undershoots true K by a factor of ~500. Any physical K-bound derived from compression-ratios alone will inherit this gap.

**Finding 2 — gzip is tighter for structured content, LZ78 is alphabet-calibrated for random content**: The two proxies are complementary. Taking `min(gzip_ratio, lz78_norm_corrected)` as the K-estimate would improve the bound for structured strings. But neither bound helps for globally-algorithmic strings.

**Finding 3 — the missing ingredient for R1 is program-length**: The only way to tighten the K bound for math constants and LCG strings is to account for the length of the generating program. Compression-based proxies cannot do this. A physical K-bound that only uses the holographic bound (which counts distinguishable states = S-information) will correctly bound the S-side but will be vacuous for the K-side when the string is globally-algorithmic with short description.

**Finding 4 — alphabet size is a physical degree of freedom**: LZ78 phrase counts scale with log(|Sigma|). For a physical system, the effective alphabet is the number of distinguishable microstates per site, which is bounded by the holographic area law (S ≤ A/4 in Planck units). A physical LZ78-like bound would be `K ≥ n * log(d) / log(n)` where d is the local Hilbert space dimension. This connects the K-proxy to a genuine physical quantity (local state-space dimension), partially answering R1 for the compression-proxy lower bound.

**Residual**: R1 remains open for the upper bound on K (how much structure can be packed into n bits in a given region). The numerical work here characterizes the lower-bound side only.
